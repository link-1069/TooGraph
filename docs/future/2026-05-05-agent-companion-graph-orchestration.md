# 桌宠 Agent、自我设定与图操作权限方针

## 文档优先级

本文以 2026-05-05 关于桌宠权限、记忆、人设和图操作边界的讨论为第一优先级。旧版“桌宠 Agent 与自动编排图设想”中的视觉、交互、自动编排、GraphCommandBus 等内容只作为补充信息保留；如果旧设想和本文规则冲突，以本文为准。

核心结论：

- 记忆、人设、语气、沟通偏好、行为边界属于桌宠自我设定，在任何档位都可以由用户编辑。
- 桌宠自我设定和记忆由图模板或 skill 驱动更新，后端只提供存储、revision、校验和读取上下文等原语；第一版可以是后端 JSON 文件，但不走前端 localStorage。
- 自动记忆是桌宠核心能力，不提供开关，也不在每次写入前询问用户；用户通过侧边栏 Companion 页面查看、编辑、删除和恢复。
- 每一次自我设定、记忆和会话摘要更新都必须先备份旧值，形成可追溯、可恢复的 revision。
- 三档权限只控制桌宠是否能操作 GraphiteUI 的图，包括新建图、修改图、连接节点、删除节点、应用补丁、运行图等。
- 桌宠自我设定不能覆盖系统级安全规则，也不能把当前图操作档位升级到更高权限。
- 图操作必须由程序能力门禁控制，不能只依赖 prompt 约束。

## 两条独立权限线

桌宠系统必须把 `selfConfig` 和 `graphOps` 分开实现。

`selfConfig` 是桌宠自我设定能力：

- 读取和编辑桌宠名字、人设、语气、回应风格。
- 读取和编辑长期记忆、用户偏好、桌宠行为边界。
- 读取和编辑会话摘要、可见的记忆条目和可删除的记忆记录。
- 读取历史版本，并从历史版本恢复当前设定。
- 在建议档、审批档、全权限档都默认可用。

`graphOps` 是图操作能力：

- 解释当前图、读取当前图结构、读取校验结果。
- 生成图修改建议、图补丁或新图草案。
- 应用图补丁、创建节点、连接边、删除节点、修改配置。
- 新建图、保存图、运行图、根据运行反馈继续修改图。
- 随三档权限变化。

用户可编辑“行为边界”，但这只是桌宠产品设定，例如“少废话”“多解释原因”“不要主动打扰”“回答前先复述理解”。它不能覆盖 GraphiteUI 的系统级规则，例如当前档位、图命令门禁、校验器、审计、撤销、停止按钮和不可越权规则。

## 三档图操作权限

### 建议档

建议档是第一阶段能力。桌宠可以聊天、解释当前图、分析运行错误、提示校验问题、提出编辑建议，也可以编辑自己的记忆、人设、语气和行为边界。

程序上，建议档不暴露任何图写入能力：

- 不能新建图。
- 不能修改当前图。
- 不能创建节点、连接边、删除节点。
- 不能应用图补丁。
- 不能自动运行图。

建议档可以读取当前图结构和运行反馈，但只能返回解释和建议。它给出的内容应该是用户可理解的自然语言建议，必要时可以附带“如果进入审批档，可以生成一个草案”的提示。

### 审批档

审批档允许桌宠根据用户需求生成图修改方案或新图方案，但必须经过用户确认后才能应用。

审批档的图操作流程应为：

1. 用户提出目标。
2. 桌宠读取当前图、模板、校验结果和必要上下文。
3. 桌宠生成图草案或图补丁。
4. GraphiteUI 以预览层、diff 或草案视图展示变更，不直接写入当前图。
5. 用户确认后，补丁才进入命令系统。
6. 命令系统应用变更，并生成审计记录、撤销点和校验结果。

审批档下，记忆、人设、语气和行为边界仍然可以直接编辑，不需要因为图操作档位是审批档而额外审批。

### 全权限档

全权限档允许桌宠直接新建图、修改图、连接节点、删除节点、保存图、运行图，并根据运行反馈继续调整图，不需要人类逐步确认。

全权限档不是绕过工程系统。它仍然必须：

- 通过 GraphiteUI 的命令系统操作图，而不是直接改 DOM。
- 通过图校验器。
- 记录审计日志。
- 生成可撤销操作。
- 支持用户随时暂停、停止或接管。
- 在 UI 上让用户看见桌宠正在做什么、为什么这么做、下一步准备做什么。

全权限档的“不加限制”指不需要人类审批每一步图操作；它不表示可以绕过程序边界、校验器、审计、撤销和停止机制。

## 记忆、人设与上下文系统

具体第一阶段落地设计见 `2026-05-05-companion-self-config-memory-design.md`。本节保留系统方针。

参考 `demo/claude-code-source` 和 `demo/hermes-agent` 的实现，桌宠不应把人设、长期记忆、当前会话和工具权限混成一段 prompt。推荐拆成四类状态：

- `companion_profile`：桌宠名字、人设、语气、展示偏好、默认回应风格。
- `companion_policy`：当前图操作档位、可用图命令、哪些图操作需要审批、哪些操作永远禁止。
- `companion_memory`：长期记忆，保存用户偏好、桌宠反馈、项目背景、外部资料指针、图设计约定等不可从当前图直接推导的信息。
- `companion_session_summary`：当前对话的压缩摘要，保存本轮目标、已确认决策、待处理问题、当前图状态摘要。

记忆系统应遵循两个原则。

第一，长期记忆采用“索引 + 详情”的结构。索引只保留标题、类型、简短说明和更新时间，详细内容另存。召回时先根据索引选择少量相关记忆，再把详情作为只读背景注入当前请求。

第二，召回记忆临时注入，不污染原始对话。召回文本应当用类似 `<memory-context>` 的边界包裹，并明确说明它不是新用户输入。它可以帮助桌宠理解当前请求，但不应该写回用户可见消息，也不应该被当成新的高优先级指令。

第三，所有自我设定和记忆更新必须通过指定图模板、skill 或同等可审计命令路径写入，并创建 revision。用户不需要在每次记忆前确认，也不需要开关；透明性通过侧边栏 Companion 页面、历史版本、删除和恢复能力实现。

## 记忆压缩策略

记忆压缩分为会话压缩和长期记忆整理。

会话压缩面向当前对话窗口：

- 保留最新用户请求、最近几轮对话和当前图状态。
- 把中间历史压成结构化 `companion_session_summary`。
- 摘要只能作为参考，不能成为新的系统规则。
- 摘要中不保存内联媒体载荷、大块媒体内容或可以从当前图重新读取的信息。

长期记忆整理面向跨会话记忆：

- 只保存未来仍有价值、稳定、非临时的信息。
- 适合保存用户偏好、反复出现的反馈、项目背景、外部资料指针、图设计习惯。
- 不保存当前图里已经能读到的节点结构、一次性运行细节、临时任务状态、代码路径或可从项目文件直接推导的信息。
- 记忆可以被用户查看、编辑、删除和从历史版本恢复。

第一阶段使用后端 JSON 作为 profile / policy / memory / session summary 与 revisions 的存储原语，并用图模板或 skill 表达自我设定更新、记忆整理和会话压缩行为；后续再迁移到数据库、检索、审计和更完整的可视化管理。

## 桌宠对话循环

第一档桌宠对话循环应保持只读图能力和可写自我设定能力：

1. 接收用户消息。
2. 读取 `companion_profile`、`companion_policy`、相关 `companion_memory`、当前 `companion_session_summary`。
3. 读取当前页面、选中节点、当前图结构、校验结果和运行反馈。
4. 根据当前档位注册可用能力。
5. 在建议档，不注册任何图写入工具，只允许返回解释、建议和自我设定修改。
6. 如果用户要改记忆、人设或行为边界，走 `selfConfig` 写入路径。
7. 如果用户要求改图，建议档只能给建议；审批档生成预览补丁；全权限档通过命令系统直接执行。
8. 回合结束后更新会话摘要，必要时整理长期记忆。

这条循环的关键是：桌宠是否能写自我设定，和它是否能写图，是两套不同门禁。

## 视觉与交互补充

旧版文档中的视觉设想可以保留为补充方向，但不改变权限规则。

桌宠是一个漂浮在 GraphiteUI 画布或侧栏附近的小型角色。它不是简单装饰，而是一个低干扰协作入口：

- 用户可以点击它发起对话、请求解释当前图、请求优化建议或请求生成图草案。
- 它可以根据当前图状态、用户最近操作、运行错误、校验结果主动给出建议。
- 在对应图操作档位允许时，它可以调用 GraphiteUI 的编辑命令操作画布。
- 它可以通过动画状态表达待机、观察、思考、建议、执行、等待确认、运行中、完成、失败。

桌宠默认保持低存在感，不遮挡节点、端口和线条。它可以停靠在右下角、右侧栏边缘或画布空白处。用户可以拖动它的位置，也可以折叠。

主动问候应当有节制。同一会话内应限制次数，且必须结合上下文，例如：

- “这个图现在有 2 个 state 没有被任何节点引用，要不要我帮你检查？”
- “当前输出节点还没有绑定输入 state，运行后可能没有可预览结果。”
- “你刚创建了条件节点，要不要我帮你补 true/false/exhausted 三条路线？”

视觉实现可以分阶段：

- 第一阶段使用现有 public mascot 素材或轻量 CSS 动画，实现拖拽、待机、说话和思考状态。
- 第二阶段升级为 SVG 或组件化状态机，让表情和动作可维护。
- 第三阶段再考虑少量 Web Animations 或 Canvas 增强效果，但核心交互仍应保持可访问、可命中、可主题化。

## GraphCommandBus 补充设想

如果后续开放审批档或全权限档，桌宠不应该直接操作 DOM，也不应该模拟点击。正确方式是新增或复用明确的编辑命令接口，让桌宠调用和用户操作同级的 GraphiteUI 能力。

建议抽象为 `GraphCommandBus`：

- `createNode(kind, position, preset?)`
- `updateNode(nodeId, patch)`
- `connectState(sourceNodeId, sourcePort, targetNodeId, targetPort)`
- `connectFlow(sourceNodeId, targetNodeId, route?)`
- `removeEdge(edgeId)`
- `focusNode(nodeId)`
- `openPanel(panelId)`
- `validateGraph()`
- `runGraph()`
- `saveGraph()`

所有命令都应该经过同一套事务系统，支持撤销、重做、审计和失败回滚。桌宠每次修改图都应该形成一条可读操作记录，例如“创建 agent 节点 answer_helper”“连接 question 到 answer_helper”“将 answer 绑定到 output_answer”。

## 自动编排图补充设想

自动编排图是未来能力，不是第一阶段默认能力。它应映射到三档权限：

- 建议档：不自动编排，只提供自然语言建议。
- 审批档：生成草案或补丁，在预览层展示，用户确认后应用。
- 全权限档：边走边画，自主创建、校验、运行和迭代。

### 审批档：规划后等待人类确认

流程是：

1. 用户描述任务目标。
2. 桌宠分析任务，生成节点规划，包括输入节点、Agent 节点、条件节点、输出节点、需要的 state、数据流和顺序流。
3. 桌宠生成图草案 JSON 或图补丁。
4. GraphiteUI 在预览层高亮展示新增节点和连接，桌宠解释每个节点的作用。
5. 用户检查、拖动、编辑、删除或拒绝草案。
6. 用户确认后，GraphiteUI 才把补丁应用到当前图。

这个模式的关键是“桌宠负责起草，人类负责确认”。预览不是写入，确认才是写入。

### 全权限档：边走边画的自主编排

流程是：

1. 用户给出目标，并把桌宠切换到全权限档。
2. 桌宠创建最小可行图，例如输入节点、一个 Agent 节点、输出节点。
3. 桌宠根据校验结果、运行反馈和中间结果逐步添加节点。
4. 画布上的节点持续变化：新增节点出现、线条连接、条件分支补齐、输出绑定更新。
5. 桌宠记录每一步为什么这么做。
6. 当桌宠认为图已经能实现目标时，自动保存并运行图。
7. 图运行结束后，桌宠通知用户结果，并提供查看运行详情、继续优化或回退修改的入口。

这个模式的关键是“桌宠自己走一步看一步”。它必须建立在命令系统、校验器、撤销、审计、运行反馈和停止机制可靠的基础上。

## 分阶段落地建议

第一阶段：建议档桌宠。

- 顶层可拖拽桌宠。
- 不需要圆形窗口和小点提示。
- 支持打开对话面板。
- 侧边栏提供独立 Companion 页面，支持编辑桌宠人设、记忆、语气和行为边界。
- 后端提供自我设定和记忆的存储原语，每次通过模板或 skill 更新时创建 revision。
- 自动整理长期记忆，不询问用户是否记住，也不提供关闭开关。
- 能读取当前图、选中节点、校验结果和运行反馈。
- 只能解释和建议，不能修改图。

第二阶段：自我设定与记忆管理。

- 增加 profile / memory / session summary 的可视化管理。
- 支持查看、编辑、删除长期记忆。
- 支持会话摘要压缩。
- 支持记忆召回的只读注入。

第三阶段：审批档图草案。

- 支持生成图草案或补丁。
- 支持预览层、diff、确认后应用。
- 支持每次应用生成审计记录和撤销点。

第四阶段：全权限档自主编排。

- 支持直接通过命令系统修改图。
- 支持自动校验、运行、根据反馈迭代。
- 支持停止、暂停、接管、回退。

## 长期目标

长期看，桌宠 Agent 可以成为 GraphiteUI 的协作层。用户不需要在空白画布上从零思考每一个节点，而是可以先描述目标，由桌宠起草结构，再由用户审查和微调。对于复杂任务，桌宠也可以在全权限档下逐步扩展图，把自己的计划和执行过程可视化。

最终体验应该是：用户能随时看见桌宠正在做什么、为什么这么做、下一步准备做什么，并且能随时暂停、修改或接管。GraphiteUI 负责提供可见、可控、可验证的图编辑环境；桌宠负责把自动化能力变成一个可理解的协作者。
