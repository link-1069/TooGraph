# 控制流 / 数据流双平面协议设计

## 目标

将 GraphiteUI 的节点系统收口成更接近 LangGraph 原生心智的协议：

- 上下表达控制流
- 左右表达数据流
- 控制流边为正式运行语义
- 数据流线为自动投影语义

本设计的目标不是单纯调整 UI，而是统一：

1. 用户心智
2. 前端交互
3. 正式图协议
4. 后端运行语义

## 设计结论

### 1. 正式保存的边

正式图结构中只保留控制流边：

- `edges`
  - 表示普通控制流
  - 含义是“当前节点执行完成后，进入下一个节点”
- `conditional_edges`
  - 表示条件路由
  - 含义是“当前条件节点根据 branch 选择下一个节点”

普通 `edges` 不再承担数据端口语义。

也就是说，后续正式协议中的普通边应逐步从：

```json
{
  "source": "a",
  "target": "b",
  "sourceHandle": "write:answer",
  "targetHandle": "read:answer"
}
```

收口成只表达控制流的形式：

```json
{
  "source": "a",
  "target": "b"
}
```

### 2. 自动投影的数据流线

左右两侧的数据流线不进入正式协议，不直接参与运行。

它们由以下信息自动推导：

- `state_schema`
- `nodes[*].reads`
- `nodes[*].writes`
- 当前控制流拓扑

数据流线的职责只有一个：帮助用户理解 state 从哪里来、被谁读取。

### 3. 控制流可编辑，数据流不可编辑

用户可以直接编辑的只有控制流：

- 顶部流程输入
- 底部流程输出
- 普通控制流边
- 条件 branch 控制流边

用户不能直接编辑数据流线。

数据流线只在以下操作中被间接改变：

- 修改节点的 `reads / writes`
- 修改 state
- 修改控制流拓扑
- 新建节点时自动补入必要的 state 绑定

## 节点协议

### input

- 顶部：无流程输入
- 底部：1 个流程输出
- 左侧：无数据输入
- 右侧：1 个数据输出

语义：

- `input` 是流程起点之一
- `input` 将 boundary value 写入 graph state
- `input` 可以继续驱动后续节点执行

### agent

- 顶部：1 个流程输入
- 底部：1 个流程输出
- 左侧：数据输入（reads）
- 右侧：数据输出（writes）

语义：

- `agent` 的执行顺序由控制流决定
- `agent` 的输入输出 state 由 `reads / writes` 决定

### condition

- 顶部：1 个流程输入
- 底部：多个 branch 流程输出
- 左侧：数据输入（reads）
- 右侧：空

语义：

- `condition` 是控制流路由节点
- 每个 branch 在底部以独立出口呈现
- branch 名称展示和双击编辑，都放在底部 branch 出口上
- 右侧不再承担 branch 的连接或展示职责

### output

- 顶部：1 个流程输入
- 底部：无流程输出
- 左侧：数据输入（reads）
- 右侧：空

语义：

- `output` 是终点节点
- `output` 不承担透传职责
- 如果未来需要“中间查看后继续”，应单独设计新的中间节点，而不是让 `output` 继续出流程

## 条件节点细则

### 默认分支

新建 `condition` 节点时，默认创建两个 branch：

- `continue`
- `retry`

默认映射：

- `true -> continue`
- `false -> retry`

### branch 编辑

双击底部 branch 项时，打开小面板编辑：

- branch 名称
- branch mapping

branch mapping 的当前阶段语义只处理双分支布尔场景，不扩展到更复杂的 switch-case 设计。

### 循环上限

循环上限挂在 `condition.config.loopLimit` 上，而不是图级 metadata。

语义：

- `5` 表示该条件节点控制的回环最多允许进入第 5 次，第 6 次失败
- `-1` 表示无限制

## 数据流投影规则

### 基本规则

对某个节点 `N` 的某个 `read state = S`：

1. 在当前控制流图中，寻找所有能够到达 `N` 的 writer(`S`)
2. 过滤掉那些在到达 `N` 之前会被更近 writer 覆盖的候选
3. 将最近的有效 writer 作为 `N` 的数据来源

投影线只用于展示：

- 哪个上游节点为这个 read 提供了当前最可能的数据来源

### 当前阶段约束

当前阶段允许：

- 同一个 state 被多个节点顺序写入
- 但这些 writer 必须处于明确的控制流顺序中

例如：

```txt
A -> B -> C -> D

B 写 answer
C 再写 answer
D 读 answer
```

则：

- `D` 读到 `C` 写入后的最新值
- 数据流投影显示 `C -> D`

当前阶段不支持：

- 并列无序 writer 同时写同一个 state
- 并列 branch 汇合后，对同一个 state 的多候选静态来源
- reducer 型 state 合并

也就是说，下列场景应在校验阶段报错，而不是尝试投影：

```txt
      -> B 写 answer ->
A ->                    -> D
      -> C 写 answer ->
```

## 创建节点时的交互规则

### 从数据口拖出到空白画布

允许从左右数据口拖到空白处创建节点。

此时：

1. 自动创建新节点
2. 自动补充兼容的 `read` 或 `write`
3. 自动补一条控制流边，使新节点进入当前流程

这里的关键点是：

- 真正保存的是控制流边
- 数据线只是因为 `reads / writes` 改变而自动重新投影

### 从流程口拖出到空白画布

允许从顶部/底部流程口拖到空白处创建节点。

此时默认行为：

1. 自动创建新节点
2. 自动补充普通控制流边
3. 不强制自动绑定 state

仅当下面条件同时满足时，允许自动推断一个默认 state 绑定：

- 源节点只有一个明显主写入 state
- 新节点只有一个明显兼容的默认输入 state

否则不做猜测。

## 线条显示规则

### 控制流边

控制流边是真正可编辑的线：

- 使用节点顶部 / 底部流程口连接
- 走正交圆角线
- 条件 branch 的线使用更强路由语义样式

### 数据流线

数据流线是自动演算的弱化线：

- 只表达 state 来源与读取关系
- 不可直接编辑
- 不保存进正式协议
- 必须与控制流边在样式上明显区分

## 对正式协议的影响

这套设计最终要求协议边界变成：

### 正式运行语义

- `state_schema`
- `nodes`
- `edges`
- `conditional_edges`

### 自动投影语义

- 数据流线
- 数据端口之间的来源指向

这意味着：

- 前端当前把普通边写成 `write:state -> read:state` 的做法需要逐步收口
- 后端普通 `edges` 的校验与运行语义也需要逐步从“数据端口一致”转成“纯控制流”

## 非目标

本阶段不处理：

- reducer 语义
- 并列多 writer 合并
- switch-case 多分支复杂映射
- `output` 透传
- 数据流线的手动编辑

## 实施优先级

建议后续实现顺序：

1. 重定义节点的上下/左右协议
2. 将 `condition` 的 branch 控制口改到底部
3. 将普通 `edges` 收口为控制流边
4. 新增数据流自动投影层
5. 增加多 writer 顺序覆盖校验
6. 删除旧的“普通边兼做数据边”的前后端逻辑
